### 2.3 Работа со сделками
Функции:
- Создание запроса на покупку
- Подтверждение сделки продавцом
- статуса объявления (active → reserved → sold)
- Завершение сделки
---
### model
### `DealStatus enum`
```java
public enum DealStatus {
    PENDING,    // Запрос создан, ожидает подтверждения
    APPROVED,   // Продавец подтвердил, товар зарезервирован
    COMPLETED,  // Сделка завершена, товар продан
    CANCELLED   // Сделка отменена
}
```
### `Status enum` (обновлен)
```java
public enum Status {
    Active,
    Reserved,  // Новый статус для зарезервированных товаров
    Sold
}
```

### `Deal` entity
- `id`- Long - id сделки
- `listing`- Listing - Связанное объявление (ManyToOne)
- `buyer`- User - Покупатель (ManyToOne)
- `seller`- User - Продавец (ManyToOne)
- `status`- DealStatus - Статус сделки (PENDING/APPROVED/COMPLETED/CANCELLED)
- `createdAt`- LocalDateTime - Дата создания
- `updatedAt`- LocalDateTime - Дата последнего изменения
- `completedAt`- LocalDateTime - Дата завершения (если сделка завершена)

**Связи:**
- `@ManyToOne` с `Listing` - много сделок может быть связано с одним объявлением
- `@ManyToOne` с `User` (buyer) - много сделок у одного покупателя
- `@ManyToOne` с `User` (seller) - много сделок у одного продавца

## service
### `DealService`
Бизнес-логика для работы со сделками:
#### 1. `createPurchaseRequest` - создание запроса на покупку
- Проверка существования покупателя и объявления
- Проверка статуса объявления (только Active)
- Проверка, что покупатель не является владельцем
- Проверка отсутствия активной сделки по объявлению
- Создание сделки со статусом PENDING
#### 2. `confirmDeal` - подтверждение сделки продавцом
- Проверка, что подтверждение выполняет продавец
- Проверка статуса сделки (только PENDING)
- Обновление статуса сделки на APPROVED
- Обновление статуса объявления на Reserved
#### 3. `completeDeal` - завершение сделки
- Проверка, что пользователь является участником сделки
- Проверка статуса сделки (только APPROVED)
- Обновление статуса сделки на COMPLETED
- Обновление статуса объявления на Sold
- Установка даты завершения
#### 4. `cancelDeal` - отмена сделки
- Проверка, что пользователь является участником сделки
- Запрет отмены завершенной сделки
- Если сделка была APPROVED - возврат объявления в Active
- Обновление статуса сделки на CANCELLED
#### 5. `getDealById` - получение сделки по id
- Проверка, что пользователь является участником сделки
#### 6. `getUserDeals` - получение всех сделок пользователя
- Поддержка пагинации и сортировки
- Возврат только сделок текущего пользователя
#### 7. `convertToDealResponse` - Конвертер с Deal в DealResponse
- Преобразование сущности в dto для отправки клиенту

## controller

### `DealController`
REST API endpoints для работы со сделками:
- `POST`  `/api/deals/purchase-request` - Создание запроса на покупку. Только авторизованные
- `PUT`  `/api/deals/{dealId}/confirm` - Подтверждение сделки продавцом. Только продавец
- `PUT`  `/api/deals/{dealId}/complete` - Завершение сделки. Только участники
- `PUT`  `/api/deals/{dealId}/cancel` - Отмена сделки. Только участники 
- `GET`  `/api/deals/{dealId}` - Получение сделки по id. Только участники
- `GET`  `/api/deals/user` - Все сделки пользователя. Только авторизованные |

**Параметры пагинации для `/user`:**
- `page` - номер страницы (по умолчанию 0)
- `size` - размер страницы (по умолчанию 10)
- `sortBy` - поле сортировки (по умолчанию createdAt)
- `direction` - направление сортировки (asc/desc, по умолчанию desc)
