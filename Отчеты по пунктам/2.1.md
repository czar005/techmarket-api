# Отчет по блоку 2.1: Аутентификация и авторизация

Рассказываю, что сделал, как это работает и как проверить.

---

## Что нужно было сделать

По заданию требовалось:
1. Регистрация новых пользователей
2. Вход в систему (логин)
3. Генерация JWT-токенов (это такие "пропуска")
4. Разграничение прав для обычных юзеров и админов
5. Защита эндпоинтов, чтобы просто так никто не лазил

---

## Что я сделал

### 1. Модель пользователя
Добавил таблицу `users` в БД. У юзера есть:
- email (логин)
- пароль (хранится в зашифрованном виде)
- роль (USER или ADMIN)

### 2. Регистрация и вход
Написал два эндпоинта:
- `POST /api/auth/register` — создание нового аккаунта
- `POST /api/auth/login` — вход, получение токена

### 3. JWT-токены
При успешном входе сервер отдает токен. Его нужно сохранить и отправлять с каждым запросом в заголовке:
```
Authorization: Bearer <сам_токен>
```

### 4. Безопасность
Настроил Spring Security так, что:
- К регистрации и логину можно обращаться без токена
- Ко всем остальным эндпоинтам — только с токеном
- Роли проверяются (админам можно больше)

### 5. Обработка ошибок
Сделал понятные ответы, если что-то пошло не так:
- Email уже занят → 409 Conflict
- Неверный пароль → 401 Unauthorized
- Пользователь не найден → 404 Not Found

## Как это работает (для чайников)

**Представь, что наш бэкенд — это закрытый клуб.**

1. Ты приходишь первый раз — нужно **зарегистрироваться** (оставить email и пароль на входе). Тебе говорят: "Ок, записал".
2. Когда хочешь зайти — **логинишься**. Отправляешь email и пароль, получаешь **пластиковую карту (JWT-токен)**.
3. Чтобы попасть в разные комнаты (эндпоинты типа `/listings`), ты **показываешь карту** в специальном окошке (заголовок Authorization).
4. Если карта просрочена или левая — охрана (наш бэкенд) не пускает и выдает ошибку.
5. У админов карта с золотым покрытием — их пускают в ВИП-комнаты (эндпоинты для админов).

## Как я тестировал

### Инструменты
- Postman (для ручных тестов)
- Юнит-тесты (JUnit) — но пока базовые

### Что проверял

**1. Регистрация нового юзера**
```
POST /api/auth/register
{
    "email": "student@example.com",
    "password": "qwerty123",
    "role": "USER"
}
```
→ Получил токен, статус 200 OK.

**2. Регистрация с тем же email**
```
POST /api/auth/register (с тем же email)
```
→ Получил ошибку 409 Conflict с сообщением "User already exists".

**3. Логин с правильным паролем**
```
POST /api/auth/login
{
    "email": "student@example.com",
    "password": "qwerty123"
}
```
→ Получил новый токен.

**4. Логин с неправильным паролем**
```
POST /api/auth/login
{
    "email": "student@example.com",
    "password": "wrong"
}
```
→ Ошибка 401 Unauthorized.

**5. Доступ к защищенному эндпоинту**
```
GET /api/test/user
Authorization: Bearer <токен_юзера>
```
→ Получил "Доступно USER" (работает).

**6. Проверка ролей**
- С токеном USER пытаюсь зайти на `/api/test/admin` → 403 Forbidden (не пускает)
- С токеном ADMIN захожу на тот же эндпоинт → успех

## Важно для остальных

### Если будете использовать мою часть:

1. **Для запросов, где нужна авторизация**, обязательно добавляйте заголовок:
   ```
   Authorization: Bearer <токен>
   ```
   Без него будет 401 Unauthorized.

2. **Роли работают так:**
   - USER — обычный пользователь (может работать только со своими объявлениями)
   - ADMIN — админ (может всё)

3. **Примеры запросов** лежат в папке `postman/` (я экспортировал коллекцию).

4. Если что-то непонятно с ответами — ошибки теперь приходят в формате:
   ```json
   {
     "status": 409,
     "message": "User with email test@example.com already exists",
     "timestamp": 1678901234567
   }
   ```
